Core architecture decision for [[stealth-agent-glasses-product]] rendering (the [[project-goal-4dgs]] side): **generation is NOT a runtime step.** Build a growing library of pre-made (and pre-rigged where needed) assets; at runtime do RETRIEVAL, not generation. This is a network-effect **data moat** (library compounds with usage → marginal render cost → ~0).

**Runtime:** input (snapshot/prompt) → embed (CLIP/DINO) → vector NN search w/ similarity score → HIT: fetch+render+animate on phone → display. MISS: render closest match as placeholder + queue background generation (result lands in library for next time).
**Offline (cloud GPUs):** generate/source → mesh → [auto-rig IF animatable] → [motion] → compress → embed → store.

**Key refinements / risks:**
- **Split library: STATIC props vs ANIMATABLE entities.** Most everyday objects (bottle, cup, chair, table, TV, car, menu) are RIGID — no skeleton/AnimaX/rigging needed, just a static asset. Rigging+motion only for humans/animals/characters (agent persona, remote person, pet). Don't waste rigging effort on cups.
- **Placeholder-swap UX risk is AR-specific:** overlaying a "similar" generic asset onto the user's REAL object (registered to reality) is uncanny in a way flat-screen apps aren't. Only overlay when no real referent (e.g. shopping "show in my room"), or ghost+snap when ready.
- **Storage:** global library is cloud/CDN; device caches PREDICTIVELY via agent context (location/time) — context engine = the cache prefetcher.
- **"Costs nothing" caveat:** marginal cost →0 (the moat is real), but standing CDN/storage/retrieval + cold-start gen cost remain. Commercial library needs clean asset provenance (Wan2.1/CC-BY/CC0 stack from [[animax-reimplementation-findings]]).
- **Cold-start fix:** seed the library offline from **Objaverse-XL (~10M objs) / ShapeNet** converted to our format — launch with the distribution head already cached instead of empty; user data grows the tail.
- **Multi-view capture:** on a generation miss, exploit that the glasses see a VIDEO stream — capture multiple frames as the user looks around the object → multi-view reconstruction, far better than single-snapshot→3D (which hallucinates unseen sides).
- **Library janitor:** every miss minting a new asset → thousands of near-dup chairs of varying quality. Needs dedup (merge near-neighbors) + quality-scoring (keep best per cluster) or retrieval DEGRADES with growth. Self-improving requires self-pruning.
- **User-captured brand IP / privacy:** redistributing assets generated from user snapshots of branded real objects (McDonald's, specific cars) to all users = trademark/likeness + privacy exposure (uploading images of people's homes). Build consent + brand-filtering into capture, not after.

**Data model — class (geometry) vs instance (appearance):** normalize assets as a **class** (canonical mesh + rig + physics/collision params + category + geometry-embedding) with many **appearance instances** (splat blob + appearance-embedding + optional brand/price/SKU metadata + owner: user_id for private | null for shared/brand). Retrieval is two-stage coarse-to-fine: match class by geometry, then instance by appearance. ONE elegant payoff: the same model expresses both a generic everyday object AND a branded commerce SKU (Nike Air Max = 1 class, 12 colorway instances + price metadata).
- **Win is storage + authoring, NOT frame cost:** sharing a class dedups geometry and reuses the expensive rig/physics authoring; per-frame render still rasterizes N splats (sharing doesn't make the frame cheaper).
- **CRITICAL constraint:** splats are bound to mesh triangles, so "same mesh, different splat" only transfers cleanly when variation is APPEARANCE-ONLY (Doritos flavors, book covers, colorways). For genuinely variable geometry (beds, furniture, organic) the user's bed example is the BREAKING case — their splat on a generic mesh smears where geometry disagrees. Fallbacks: canonical-splat (generic look), parametric/morphable class template (SMPL-style, few categories exist, v2 research), or re-bind=generation. For STATIC props (most everyday items, no animation) skip shared mesh entirely: store per-instance splats + cheap collision proxy (bbox/hull); "class" is then just a retrieval/dedup grouping. Mesh-sharing's real payoff is ANIMATABLE classes (rig genuinely reused).
- **Privacy-by-decomposition (user's idea, correct posture):** shared library = generic/non-personal geometry; private = per-user appearance splat stored to their profile, never enters shared library. Caveat: the personal splat is still personal data (siloed, NOT anonymized — label it right). Requires object segmentation at capture (don't vacuum up the whole room).
- **New hard problem:** class-assignment threshold ("reuse class mesh vs mint new class") = the dedup/janitor threshold pushed up into geometry space, two-tiered with the appearance threshold. This is where the design needs tuning.
- **Build order for this model:** rigid-class path first (packaged goods, books, branded SKUs = the commerce wedge); parametric morphable classes are v2.

**Three scopes = ONE pool, two metadata flags (not three DBs):** visibility = global | marketplace | private; curation = hand-curated | provider-verified | user-generated. Global=(global,hand-curated) generic objects; Marketplace=(marketplace,provider-verified) buyable products w/ commerce metadata, providers self-upload (→ needs verification + quality gating + moderation = a UGC/trust layer); Private=(private,user-generated) user scans/creations, premium-tier feature. Dedup falls out for free: on scan, "does a matching class exist in the user's visible scope?" → reuse it, don't mint a private duplicate.
- **THREE-level taxonomy (fix the "chair is a class" bug):** Category (semantic, for browse/search — "chair") > Class (shared mesh+rig+physics — "wooden 4-leg dining chair") > Instance (appearance splat). User's "versions of a chair" (rocking chair, couch, stool) are SEPARATE CLASSES under the chair category, NOT instances of one class — couch/rocker have different geometry+physics, can't share a mesh.
- **Class match ≠ appearance shared:** when a user's scan matches a shared class, reuse the class (mesh/rig/physics) BUT their captured splat stays a PRIVATE instance pointing at that shared class. Shared geometry, private appearance. Don't promote their capture to global, don't discard their look.
- **STRATEGIC FORK (unresolved → I recommended consent-gated):** "manual-only global + private-never-promotes" KILLS the data-moat flywheel (library no longer grows with usage). Two options: (a) closed-private = cleaner privacy, no compounding moat, long tail cached only per-user; (b) **consent-gated promotion** = many users scan same NOVEL class → IP/privacy filter → human curation → promote the generic CLASS (never the personal splats) to global. Can't have both manual-only global AND compounding moat. Recommended (b) — only version where cost actually →0.
- **Cross-scope retrieval needs a RANKING POLICY** (business decision, not just NN): private (personalization) vs marketplace (revenue) vs global (fallback). Where personalization and monetization collide.

**SIMPLIFICATION DEBATE — "flat everything-shared + dedup-on-write, only humanoids private":** user proposed collapsing the 3 scopes to one flat shared pool: record anything → similarity-check (mesh+splat) → similar=discard, different=keep; everything shared except humanoids. My verdict — instinct partly right, but:
- **GOOD:** dedup-on-write IS a better flywheel than manual-only global (resolves the growth fork toward compounding moat); "objects shared, people private" is a clean privacy line; flatter = ships faster. The deep category/class/instance NESTING genuinely CAN be dropped for v1 (it was a search optimization, not core).
- **ILLUSION:** "no searching" is false — similarity-check IS the retrieval layer, just moved read-time→write-time where it needs HIGHER precision (a wrong discard deletes user data forever). Still need embeddings + ANN index.
- **LOAD-BEARING, CAN'T DROP:** (1) IP containment — "everything shared" = redistributing user-scanned BRANDED/copyrighted objects commercially = takedown firehose; the humanoid carve-out protects PEOPLE, not BRANDS/copyright. (2) Marketplace/commerce scope carries price/SKU/checkout = the shopping business; can't flatten it away. (3) personalization loss (focal object becomes generic).
- **SYNTHESIS (recommended):** keep the simplification but replace binary keep/discard with a write-time ROUTER (you already need a humanoid detector — extend it): person→private humanoid; branded/IP→block or private-only; buyable→marketplace; generic→flat shared dedup-on-write. Drop the nesting, keep the scopes. One classifier does it in both designs. image/text→3D = TRELLIS / Hunyuan3D-2 / Tripo (NOT LGM/InstantMesh, blobby); auto-rig = Make-It-Animatable (humanoid) / UniRig / RigAnything (general topology, imperfect); motion = AnimaX/Wan2.1; compress = Draco+meshopt+KTX2 (mesh), SOG ~100-500k splats+LOD (splat); retrieval = CLIP/DINO + vector DB.

**Build order:** v2 rigged-MESH track first (static props + animatable-mesh entities) = makes whole loop work end-to-end, cheap render; then v3 rigged-SPLAT track = representation upgrade for hero entities, research = the RUNTIME (animated splat budget + world-lock reprojection), since gen is offline.
